# 🚀 AI 협업의 진화: 프롬프트에서 하네스까지

AI와 협업하는 방식은 단순히 '질문을 잘하는 것'에서 '안전한 실행 시스템을 구축하는 것'으로 빠르게 확장되고 있습니다. 이 노트는 프롬프트 엔지니어링에서 하네스 엔지니어링으로 이어지는 진화 단계와, 거대 AI 기업(OpenAI, Anthropic)의 서로 다른 하네스 전략을 분석합니다.

---

## 1. 진화의 3단계

### 🟢 1단계: 프롬프트 엔지니어링 (Prompt Engineering)
*   **핵심:** "AI에게 말을 잘 거는 법"
*   **설명:** 모델로부터 원하는 결과를 얻기 위해 입력값(프롬프트)을 정교하게 설계합니다.
*   **한계:** 결과물의 품질이 개인의 지시 구체성에 의존하며, 맥락이 길어질수록 일관성이 저하됩니다.

### 🔵 2단계: 컨텍스트 엔지니어링 (Context Engineering)
*   **핵심:** "AI에게 지도와 참고서를 주는 법"
*   **설명:** AI가 답변하기 전에 프로젝트의 규칙(SSOT), 문서, 배경 지식을 체계적으로 주입합니다. (예: [[Obsidian_RAG_Methodology]])
*   **효과:** AI가 조직의 특수한 상황을 인지하여 답변의 정확도가 크게 향상됩니다.

### 🔴 3단계: 하네스 엔지니어링 (Harness Engineering)
*   **핵심:** "AI가 일할 수 있는 안전한 작업 시스템을 만드는 법"
*   **설명:** 작업을 그냥 실행하는 것이 아니라, 검증-계획-실행-확인 단계를 거치는 신뢰할 수 있는 인프라를 설계합니다. (예: [[Harness_Engineering]])
*   **효과:** 실수를 시스템적으로 차단하며, '혼자서도 팀처럼' 일관된 품질을 보장합니다.

## 🏆 거대 AI 기업의 하네스 전략 비교

OpenAI와 Anthropic은 하네스를 바라보는 철학에서 뚜렷한 차이를 보입니다.

| 관점 | OpenAI (하네스 엔지니어링) | Anthropic (하네스 디자인) |
| :--- | :--- | :--- |
| **핵심 은유** | **에이전트의 세계 구축** | **적대적 피드백 루프 (GAN 영감)** |
| **주요 관심사** | 환경 설계 + 대규모 코드베이스 일관성 | 장시간 자율 코딩 + 품질 보장 |
| **에이전트 수** | 단일 에이전트 (+ 리뷰 에이전트) | 3-에이전트 (Planner/Generator/Evaluator) |
| **품질 보장** | 린터, CI, 구조적 테스트, 정기 GC | 독립 Evaluator + Playwright 기반 실앱 QA |
| **컨텍스트 관리** | Progressive Disclosure | 자동 컴팩션 vs 리셋 |
| **인간의 역할** | **환경 설계자** (코드 작성 0줄 목표) | **하네스 튜너** (기준과 프롬프트 조정) |
| **엔트로피 대응** | Golden Principles + 백그라운드 정리 | 스프린트 단위 품질 게이트 (Evaluator) |

## 📊 요약 비교

| 구분      | 프롬프트 엔지니어링       | 컨텍스트 엔지니어링             | 하네스 엔지니어링                    |
| :------ | :--------------- | :--------------------- | :--------------------------- |
| **비유**  | 사서에게 말로 잘 설명하기   | 사서에게 참고 도서 목록 주기       | 사서의 **대출/반납 시스템** 구축         |
| **관심사** | 질문(Input)의 품질    | 배경 정보(Context)의 가용성    | **작업(Execution)의 안전성**       |
| **도구**  | 대화형 UI (ChatGPT) | RAG, [[_master-index]] | **gstack**, CI/CD, Validator |
| **목표**  | "좋은 답변을 받자"      | "우리 상황에 맞는 답을 받자"      | **"실패 없이 작업을 완수하자"**         |
